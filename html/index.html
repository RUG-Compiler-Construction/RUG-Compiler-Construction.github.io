RUG Compiler Construction - Shared Compiler

Building the compiler

For now, it just is a parser which will either accept or reject the syntax.

Run make or make all to build the parser. The binary is called egclc: 'Extended Guarded Command Language Compiler'.

Whenever desired, run make clean to remove all intermediate build files.

Dependencies

This project uses:

  • make and g++ which are available and pre-installed on most modern POSIX systems.
  • the LLnextgen parser generator in combination with the flex lexer/scanner generator.

Testing

Run make test to build the parser in testing mode, and test all the examples in the test/parsing directory:

  • Examples in the expression directory are expected to be parseable as an Expression.
  • Examples in the statement directory are expected to be parseable as a single Statement.
  • Examples in the statements directory are expected to be parseable as a sequence of one or multiple statements.
  • Examples in the program directory are expected to be parseable as a full Program.

This separation allows us to test parts of the parser/grammar independently, and create more concise examples for them.

For each of these directories, there is a ./counterexample subdirectory, whose contents will be expected to be rejected by the parser. This allows us to check whether the parser is strict enough. (And later on it could be extended to also test the semantic and typechecking passes after the parsing steps.)

Ast

In src/ast you can now find the code for the Abstract Syntax Tree.

  • src/ast/node/abstract contains the Base class and the Concrete templated class (no changes there).
  • src/ast/node/concrete contains all the header files *.hpp* for
    • Concrete struct definitions for the laguange nonterminals
    • Concrete struct definitions for the language leaf node (atoms, IntConstant, RealConstant, Identifier).
    • Intermediate struct definitions such as Unary, Binary or Nary
    • A file includes.hpp which includes all of the above headers.
    • A file node.hpp which includes the includes.hpp and creates the std::variant<...> with all of the concrete types.
  • src/ast/ast.hpp which as the initial implementation just includes node.hpp
  • src/ast/visitor.hpp which is the abstract implementation of the visitor pattern and should be included and extended in each of its concrete implementation.

In src there is also the examples made in the initial implementation of printing (i.e. printing/) and code generating (i.e. codegen/).

Please note that some of the nonterminals may eventually result superfluous, and some more intermediate classes could be added if needed.

Compilation is still the same as explained above: make or make test.

note

When implementing a task such as code-generation or semantic checking, the code boils down to defining the overload for the visitor of that task. It is important to remeber that at the visitior must be able to match each and every concrete class, either directly or by handling one of its superclasses.

As an example take the semantic cheking of an expression: every binary arithmentic expression will be handled that same way so probably it's enough to define the overload for an ancestory like BinaryExpression instead of each concrete.
